## Migration Notes: One-Record-Per-Respondent Schema

### Current Architecture

- **MainData**: One row per respondent — profile/demographic fields
- **AllOtherData**: EAV (Entity-Attribute-Value) — multiple rows per respondent,
  one per (QuestionNumber, Answer/Subject/Ranking/Scale) combination

### Target Architecture

One wide table with one row per respondent. All question answers become columns.

### Question Pivot Strategy

| Type | Example Questions | Column Strategy |
|------|------------------|-----------------|
| **Single-code** (one answer) | Q4, Q7, Q15, Q20a, Q20b, Q21, Q30, Q34a, Q34b, Q39, Q40, Q41, Q42, Q43, Q43a | `Q{n}` — one column, value = Answer |
| **Multi-code** (multiple answers) | Q2, Q5b, Q9b, Q10, Q11, Q13b, Q27, Q28, Q31, Q45, Q54 | `Q{n}_{answer}` — boolean column per option, OR `Q{n}` — JSON array |
| **Ranked** (ordered selections) | Q8, Q18, Q19, Q33, Q44, Q55 | `Q{n}_rank{i}` — one column per rank position |
| **Grid/scale** (rate sub-items) | Q46, Q53 | `Q{n}_{subject}` — one column per sub-item, value = Scale |
| **NPS/satisfaction** (0-10 or 1-5) | Q11d, Q29, Q40a, Q40b, Q47, Q48, Q52 | `Q{n}` — one numeric column |

### Key Decisions Needed

1. **Multi-code storage**: Boolean columns per option vs JSON array?
   Boolean is easier to query but creates many sparse columns.
2. **Ranked questions**: How many rank positions to keep? (Top 3? Top 5?)
3. **Grid questions**: Subject labels vary — need canonical naming.
4. **Cross-product consistency**: Are Motor and Home questions identical?
   If not, which columns are product-specific?
